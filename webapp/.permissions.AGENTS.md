# Permissions System Guide for AI Agents

## Overview

This application uses a **organization-scoped role-based permission system**. Every resource belongs to an Organization, and users have roles within organizations that determine their permissions.

## Architecture

### 3-Tier Permission Model

```
User ──> OrganizationMember ──> Role ──> Django Permissions
           (link)              (3 types)   (CRUD operations)
```

### Components

1. **Organization**: Container for all resources (projects, audits, etc.)
2. **OrganizationMember**: Links User + Organization + Role
3. **Role**: Defines permission level (administrator, writer, reader)
4. **Django Permissions**: Standard Django permissions for models

## Roles

### 1. Administrator
- **Full access** to all organization resources
- Can manage organization members
- Can create, read, update, delete all resources

### 2. Writer
- Can **create, read, update, and delete** resources
- Cannot manage organization members
- Cannot modify organization settings

### 3. Reader
- **Read-only** access to all resources
- Cannot create, modify, or delete anything
- Useful for auditors, external consultants

## Permission Checks

### Two-Level Verification

Every resource access checks:
1. **Organization Membership**: Does user belong to the organization?
2. **Role Permissions**: Does user's role allow this action?

## Using Permission Mixins

### In Views

```python
from organization.permissions import (
    OrganizationReadPermissionMixin,
    OrganizationWritePermissionMixin,
    OrganizationAdminRequiredMixin,
)

# Read-only access (reader, writer, or administrator)
class ProjectListView(OrganizationReadPermissionMixin, ListView):
    model = Project

# Write access (writer or administrator)
class ProjectCreateView(OrganizationWritePermissionMixin, CreateView):
    model = Project

# Admin-only access
class OrganizationSettingsView(OrganizationAdminRequiredMixin, UpdateView):
    model = Organization
```

### Available Mixins

| Mixin | Roles Allowed | Use Case |
|-------|---------------|----------|
| `OrganizationReadPermissionMixin` | All (reader, writer, admin) | Viewing resources |
| `OrganizationWritePermissionMixin` | writer, administrator | Creating/editing resources |
| `OrganizationAdminRequiredMixin` | administrator only | Admin operations |

### Automatic Features

These mixins automatically:
- Verify user authentication
- Check organization membership
- Filter querysets to current organization
- Raise `PermissionDenied` (403) if unauthorized
- Set `self.organization_member` for use in views

### Example: Using organization_member

```python
class ProjectCreateView(OrganizationWritePermissionMixin, FormView):
    def form_valid(self, form):
        project = form.save(commit=False)
        # organization_member is set by the mixin
        project.organization = self.organization_member.organization
        project.save()
        return super().form_valid(form)
```

## Session Management

### Current Organization

The active organization is stored in session:

```python
from core.middleware import CURRENT_ORGANIZATION_SESSION_KEY

current_org = request.session.get(CURRENT_ORGANIZATION_SESSION_KEY)
# Returns tuple: (org_id, org_name) or None
```

### Organization Middleware

`OrganizationMiddleware` automatically:
- Loads user's organizations into session
- Sets current organization (default or first)
- Makes organization data available in templates

## Database Schema

### Role Model

```python
class Role(models.Model):
    name = models.CharField(choices=RoleType.choices, unique=True)
    description = models.TextField()
    permissions = models.ManyToManyField(Permission)
```

### OrganizationMember Model

```python
class OrganizationMember(models.Model):
    user = models.ForeignKey(User, on_delete=CASCADE)
    organization = models.ForeignKey(Organization, on_delete=CASCADE)
    role = models.ForeignKey(Role, on_delete=PROTECT)
    is_default = models.BooleanField(default=False)
```

## Migration & Setup

### Creating Roles

Roles are created automatically by migration `0003_role_organizationmember_role.py`:

```bash
python manage.py migrate organization
```

This creates:
- `administrator` role with all permissions
- `writer` role with add/change/delete/view permissions
- `reader` role with view-only permissions

### Assigning Roles

When creating an organization:
```python
# Creator automatically gets administrator role
OrganizationMember.objects.create(
    user=user,
    organization=organization,
    role=Role.objects.get(name='administrator')
)
```

For inviting members (future feature):
```python
# Assign appropriate role when inviting
OrganizationMember.objects.create(
    user=invited_user,
    organization=organization,
    role=Role.objects.get(name='reader')  # or writer
)
```

## Security Best Practices

### 1. Always Use Mixins
```python
# ❌ Bad - No permission check
class ProjectListView(LoginRequiredMixin, ListView):
    model = Project

# ✅ Good - Organization scoped with role check
class ProjectListView(OrganizationReadPermissionMixin, ListView):
    model = Project
```

### 2. Never Trust Session Data Alone
```python
# ❌ Bad - User could manipulate session
org_id = request.session.get(CURRENT_ORGANIZATION_SESSION_KEY)[0]
project = Project.objects.get(id=project_id)  # No verification!

# ✅ Good - Mixin verifies membership
class ProjectDetailView(OrganizationReadPermissionMixin, DetailView):
    # Automatic verification that project belongs to user's organization
    model = Project
```

### 3. Filter Querysets
```python
# get_queryset() in mixins automatically filters by organization
def get_queryset(self):
    # Mixin already filtered to current organization
    queryset = super().get_queryset()
    # Add your additional filters
    return queryset
```

## Common Patterns

### Getting Current Organization

```python
def my_view(request):
    member = OrganizationMember.objects.select_related(
        'role', 'organization'
    ).get(
        user=request.user,
        organization_id=current_org_id
    )
    
    organization = member.organization
    role = member.role
```

### Checking Specific Permission

```python
class OrganizationMember(models.Model):
    # ...
    def has_permission(self, permission_codename):
        return self.role.permissions.filter(
            codename=permission_codename
        ).exists()

# Usage
if organization_member.has_permission('delete_project'):
    # Allow deletion
    pass
```

## Troubleshooting

- **403 Forbidden**: User doesn't belong to organization or lacks required role
- **No organization selected**: Session missing CURRENT_ORGANIZATION_SESSION_KEY
- **Tests failing**: Ensure fixtures create OrganizationMember with appropriate role
- **User sees empty list**: Check if organization membership exists
