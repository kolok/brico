## Default Workflow (Follow This)

1. **Explore**: Search and read the minimal set of files (views, templates, tests) to understand current behavior.
2. **Plan**: State the smallest change that satisfies the request; prefer incremental diffs.
3. **Implement**:
   - Keep changes local to the relevant app (`audits/`, `organization/`, `core/`).
   - Prefer server-rendered + Turbo/Stimulus; **do not introduce React/Vue**.
4. **Verify**:
   - Run unit tests for the touched area: `uv run pytest tests/audits/test_views.py -v`
   - Run lint/format if you changed Python: `make lint && make format`
   - If you changed TS/CSS, run frontend build: `make npm-build`

## Code Standards

### Python

- **Style**: PEP 8 via Black formatter
- **Linting**: Ruff
- **Type hints**: Required for public functions/class methods
- **Docstrings**: Required for public functions and classes
- **Naming**: `snake_case` for functions/variables, `PascalCase` for classes
- **Line length**: 88 (Black default)

Example:

```python
def calculate_audit_score(audit: Audit) -> float:
    """Calculate the overall score for an audit.

    Args:
        audit: The audit object to score.

    Returns:
        A float between 0 and 100 representing the audit score.
    """
    return (audit.points / audit.max_points) * 100
```

### TypeScript/JavaScript

- **Style**: ESLint + Prettier
- **Naming**: `camelCase` for variables/functions, `PascalCase` for classes
- **Imports**: Use modern ES6 syntax
- **Type checking**: Enable TypeScript strict mode

Example:

```typescript
export const calculateScore = (points: number, maxPoints: number): number => {
  return (points / maxPoints) * 100
}
```

### Django/DRF Patterns

- **Views**: Use class-based views when possible (APIView, ViewSet)
- **Serializers**: Keep serializers focused; use nested serializers for relationships
- **Forms**: Use Django forms for non-API endpoints
- **URLs**: Use path() with explicit view references; organize by app

### Language

- All application and documentation code is in **English**

## Commands (Use Make Targets First)

All commands run from the `webapp/` directory.

### Setup & Dependencies

```bash
make init-dev              # First-time setup: .env, Python, npm, Docker, migrations
make sync                  # Install/update Python dependencies (uv sync)
make npm-install           # Install/update npm dependencies
```

### Running the Application

```bash
make run-all              # Start Docker + Django dev server + Parcel (RECOMMENDED)
make run                  # Start Django + Parcel (requires Docker running separately)
make run-docker           # Start Docker services only
make runserver            # Start Django only (no Parcel)
```

**Development flow**:

1. `make run-all` starts everything you need
2. Access app at `http://localhost:8000`
3. Code changes auto-reload (Django + Parcel watch)
4. Stop with `Ctrl+C` in the terminal

### Database & Migrations

```bash
make makemigrations       # Create migration files from model changes
make migrate              # Apply pending migrations to database
```

**Workflow**:

1. Modify model in `models.py`
2. Run `make makemigrations` (creates file in `migrations/`)
3. Review the generated migration
4. Run `make migrate` to apply
5. Commit both the model and migration files

### Testing

```bash
make test                 # Run all tests (pytest)
uv run pytest             # Run pytest with custom args
```

**Running specific tests**:

```bash
uv run pytest tests/audits/test_views.py -v
uv run pytest tests/audits/test_views.py::test_list_audits -v
uv run pytest tests/ -k "test_create" -v

```

**Test structure**:

- Tests in `tests/` directory parallel to app structure
- Fixtures in `tests/conftest.py` or module-specific `conftest.py`
- Use descriptive names: `test_<feature>_<scenario>`
- Use `@pytest.mark.django_db` for database access

### Linting & Formatting

```bash

make lint                 # Lint Python code with Ruff
make check-format         # Check formatting without changes
```

**Before committing**:

1. `make format` – auto-fixes formatting issues
2. `make lint` – checks for issues (fix manually if needed)
3. Review changes and commit

### Frontend Assets

```bash

```

**Development**:

**Workflow**:

## Architecture & Patterns

### Backend Architecture

- **Framework**: Django 4.x with Django REST Framework
- **Database**: PostgreSQL (via Docker)
- **Package management**: `uv` (fast Python package manager)
- **API**: REST endpoints via DRF serializers
- **Frontend integration**: Hotwire (Turbo + Stimulus)

### View & API Patterns

**Class-based views** (preferred):

```python
from rest_framework import viewsets
from rest_framework.response import Response

class AuditViewSet(viewsets.ModelViewSet):
    queryset = Audit.objects.all()
    serializer_class = AuditSerializer
```

**Function-based views** (for simple cases):

```python
@require_http_methods(["GET", "POST"])
def audit_list(request):
    # Handle request
    pass
```

**Turbo frame responses** (for incremental updates):

```python
def update_audit(request, pk):
    audit = Audit.objects.get(pk=pk)
    return render(request, 'audits/detail_frame.html', {'audit': audit})
```

### Frontend Architecture

- **Framework**: Hotwire (Turbo + Stimulus)
- **Build tool**: Parcel (bundles TypeScript/CSS)
- **Source**: `static/to_compile/` (TypeScript + CSS)
- **Output**: `static/compiled/` (generated, never commit)
- **Entry point**: `static/to_compile/entrypoints/main.ts`

**Stimulus controller** example:

```typescript
import { Controller } from "@hotwired/stimulus"

export default class extends Controller {
  static targets = ["input"]

  greet() {
    console.log(`Hello, ${this.inputTarget.value}!`)
  }
}
```

**Turbo frame** example (Django template):

```html
<turbo-frame id="audit-form">
  <form method="post" action="...">
    {% csrf_token %} {{ form }}
    <button type="submit">Save</button>
  </form>
</turbo-frame>
```

### Template Patterns

- Keep templates **small and focused**; break into components in `templates/*/components/`
- Use **Django template tags** for iteration, conditionals
- Use **Turbo frames** for partial page updates (not full page reloads)
- Make **all user-facing strings translatable** via `{% trans %}` or `{% blocktrans %}`

See `./templates.AGENTS.md` for detailed template guidance.

## Quality & Conventions

### Code Quality

- **Python**: type hints required; docstrings for public functions
- **Tests**: use pytest; add/adjust tests for bug fixes & new features
- **Coverage**: keep backend coverage ≥ 80%
- **Linting**: Ruff for style, Black for formatting

### Testing Best Practices

- Use **pytest fixtures** for setup (see `tests/conftest.py`)
- Use **factories** for test data (e.g., `pytest-factory` or `factory_boy`)
- Mock external services; don't call real APIs in tests
- Test both **happy path** and **error cases**
- Use descriptive assertion messages

Example test:

```python
def test_create_audit_with_invalid_data(db):
    """Verify that invalid audit data is rejected."""
    audit = Audit.objects.create(name='', description='Test')
    assert audit.name == ''  # Or use form validation

    response = client.post('/api/audits/', {'name': '', 'description': 'Test'})
    assert response.status_code == 400
```

### i18n Conventions

- Use `{% trans "string" %}` for simple strings
- Use `{% blocktrans %}...{{ variable }}...{% endblocktrans %}` for strings with variables
- In Python: `from django.utils.translation import gettext_lazy as _; _("string")`
- Always run `make compilemessages` after translating

## Templating & UI Rules

- Prefer **Django templates + Turbo Frames** over custom JavaScript
- Keep templates small and componentized via `templates/…/components/` when appropriate
- If adding user-facing strings, make them translatable via `{% trans %}`
- Use **Stimulus controllers** for interactivity (not inline JavaScript)
- **No React/Vue frameworks** – use Hotwire

See `./templates.AGENTS.md` for component patterns and template structure.

## Safety & Repo Hygiene

### Never Commit

- **`static/compiled/`** – Generated by Parcel (add to .gitignore)
- **`.env`** – Local environment secrets
- **`__pycache__/`, `*.pyc`** – Python cache
- **`.pytest_cache/`** – Test cache
- **`.parcel-cache/`** – Parcel cache

### Before Committing

- Run `make test` – ensure tests pass
- Run `make format` – auto-fix formatting
- Run `make lint` – check for linting issues
- Run `make npm-build` – rebuild frontend if TS/CSS changed
- Commit the generated Django `.po` and `.mo` files for translations
- Keep diffs surgical; avoid broad refactors unless requested

### Commit Message Conventions

- Concise and descriptive (e.g., "Fix audit form validation" or "Add French translations for dashboard")
- Reference issue if applicable
- Follow project conventions from recent commits

## Troubleshooting

### Common Issues

**Database connection error**:

- Ensure Docker is running: `docker ps`
- Check `.env` has correct `DATABASE_URL`
- Run `make migrate` to initialize database

**Tests fail with "database locked"**:

- Stop dev servers that might hold locks
- Run `make clean` to clear caches
- Restart with `make run-all`

**Frontend doesn't update**:

- Run `make npm-watch` in another terminal
- Check for TypeScript errors in console
- Clear browser cache (Cmd+Shift+R)

**Migrations conflict**:

- Check `migrations/` directory for conflicting files
- Merge or reorder migrations if working with others
- Ask teammates for migration strategy

**Static files not found (404)**:

- Run `make npm-build` to compile assets
- Ensure `STATIC_URL` in settings points to correct location
- Check `.gitignore` isn't excluding generated files

**Memory issues with large datasets**:

- Use `select_related()` and `prefetch_related()` in querysets
- Add pagination to list endpoints
- Profile with `django-extensions` shell_plus

## Performance Considerations

- Use **select_related()** for foreign keys, **prefetch_related()** for reverse/M2M
- Add **database indexes** on frequently queried fields
- Cache expensive querysets with Django's cache framework
- Use **Turbo** for partial page updates (avoid full page reloads)
- Monitor Parcel build time; split large bundles if needed
